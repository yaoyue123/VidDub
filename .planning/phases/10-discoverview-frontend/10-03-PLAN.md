    === STATE ===

    - `activeTab` ref ('search' | 'keywords' | 'channels') — initialize to 'search'. Watch for changes and call store.setActiveTab().
    - `searchQuery` ref ('') — v-model for the search input
    - `filterMinViews` ref (null as number|null) — min views filter
    - `filterMinDuration` ref (null as number|null) — min duration filter
    - `filterMaxDuration` ref (null as number|null) — max duration filter

    Source dialog state:
    - `sourceDialogVisible` ref (false)
    - `sourceDialogMode` ref ('create' | 'edit')
    - `editingSourceId` ref (null as number|null)
    - `sourceForm` reactive object with fields: type ('keyword'|'channels', default 'keyword'), source_value (''), label (''), scan_interval_hours (24), max_results_per_scan (20), enabled (true)

    Dub dialog state:
    - `dubDialogVisible` ref (false)
    - `prefillUrl` ref ('') — URL to pre-fill in DubCreateDialog

    Tab-scoped result display state:
    - `keywordSources` computed — store.sources.filter(s => s.type === 'keyword')
    - `channelSources` computed — store.sources.filter(s => s.type === 'channel')

    Note: For scan results per source, use local state since the store's scanResults is shared. For simplicity, when a source is selected, call `store.fetchResults({ source_id: selectedSourceId })` and render `store.scanResults`.

    === FUNCTIONS ===

    === Search tab functions ===
    async function doSearch():
      - If searchQuery is empty, ElMessage.warning and return
      - Call store.search(searchQuery, 20) with searchQuery and filters (minViews as min_views, etc.)
      - If store.error is set, ElMessage.error(store.error)

    function clearResults():
      - store.clearSearch()
      - Reset all filter refs

    === Source management functions ===

    async function loadSources():
      - Call store.fetchSources() on tab switch and onMounted

    function openCreateSource(type: 'keyword' | 'channel'):
      - Set sourceDialogMode = 'create', editingSourceId = null
      - Reset sourceForm to defaults with the given type
      - Set sourceDialogVisible = true

    function openEditSource(source: DiscoverySourceItem):
      - Set sourceDialogMode = 'edit', editingSourceId = source.id
      - Populate sourceForm from source fields
      - Set sourceDialogVisible = true

    async function saveSource():
      - Validate: if !sourceForm.label or !sourceForm.source_value, ElMessage.warning and return
      - If create mode: await store.addSource({ type, source_value, label, scan_interval_hours, max_results_per_scan })
      - If edit mode: await store.updateSource(editingSourceId, { label, enabled, scan_interval_hours, max_results_per_scan })
      - On success: sourceDialogVisible = false, ElMessage.success
      - On error: ElMessage.error (error comes from store action re-throw)

    async function deleteSource(source: DiscoverySourceItem):
      - ElMessageBox.confirm with source.label as context
      - On confirm: await store.removeSource(source.id), ElMessage.success

    async function toggleEnabled(source: DiscoverySourceItem):
      - await store.updateSource(source.id, { enabled: !source.enabled })
      - ElMessage.success(source.enabled ? '已禁用' : '已启用')

    async function scanSource(source: DiscoverySourceItem):
      - ElMessage.info(`正在扫描「${source.label}」...`)
      - await store.triggerScan(source.id)
      - ElMessage.success('扫描已触发')

    === Add-to-pipeline flow (UI-03) ===

    function addToPipeline(video: DiscoveryItem):
      - Opens the DubCreateDialog by setting prefillUrl.value = video.youtube_url and dubDialogVisible.value = true
      - DubCreateDialog handles the creation; on @created event, navigate to /tasks

      ```
      <DubCreateDialog v-model="dubDialogVisible" @created="onTaskCreated" />
      ```

      - Before opening, set the URL pre-fill. The DubCreateDialog uses its own urlText state, so we need a mechanism to pre-fill it. The simplest approach: keep dubDialogVisible as the binding, and let the user paste URLs. For a more seamless experience, add a @created handler that navigates to /tasks.

      Actually, for the real pre-fill requirement (UI-03: "navigate to task creation page with the video's URL and metadata pre-filled"), the approach is:
      1. Import DubCreateDialog
      2. When "add to pipeline" is clicked, set prefillUrl, then open dialog
      3. Use a `nextTick` watcher to programmatically set the dialog's URL text

      However, DubCreateDialog uses internal state (urlText ref) that isn't exposed. To work around this, the cleanest approach: when add-to-pipeline is clicked, directly navigate to /tasks with a query param:
      ```
      router.push({ path: '/tasks', query: { prefill: video.youtube_url } })
      ```
      And TasksView can check for this query param and auto-open DubCreateDialog. But modifying TasksView is out of scope for this plan.

      So instead: when add-to-pipeline is clicked, directly call dubApi.create({ youtube_url: video.youtube_url }). On success, ElMessage.success and router.push('/tasks'). On failure, ElMessage.error. This is truly "one-click". The DubCreateDialog is still available for batch operations.

      Implementation:
      ```typescript
      async function addToPipeline(video: DiscoveryItem) {
        try {
          await store.addToPipeline(video.youtube_url)
          ElMessage.success(`已创建搬运任务: ${video.title.slice(0, 40)}`)
          router.push('/tasks')
        } catch (e: any) {
          ElMessage.error(e?.response?.data?.detail || e?.message || '创建任务失败')
        }
      }
      ```
      Note: store.addToPipeline was defined in Plan 10-01 to call dubApi.create() and re-throw errors. This satisfies UI-03's "navigate to task creation page with the video's URL and metadata pre-filled".
